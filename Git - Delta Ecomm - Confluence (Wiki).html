      Introduction

      Most of this tutorial has now been moved over to build management. You can take a look at it over in How-To: Working with Git. Here it is for reference:

      Setup

      Installation

      You can install git using mac ports. See Ecomm developer onboarding for installations instructions for mac ports. Once ports is installed, do the following in a terminal:

      ?
      # sudo port install git-core +svn +bash_completion
      # sudo port install kdiff3

      This will install git with svn support and bash completion and also install a good diff tool to use for merges.

      Since MacOS X Mavericks, you will also need to install X11 yourself: http://support.apple.com/kb/HT5293

      Configuration

      KDiff3

      Configure kdiff3 as your diff tool and set your name and email address. To do so, edit the file ~/.gitconfig. Make sure to replace the "FirstName LastName" and the email adress with your real credentials.

      ?
      [user]
              name = FirstName LastName
              email = emailaddress@nzonemedia.com
      [merge]
              tool = kdiff3
      [mergetool "kdiff3"]
              path = /opt/local/bin/kdiff3

      Bash completion

      To actually get the bash completion working, you need to add the following to your profile, e.g. ~/.profile or ~/.bashrc or whatever you use.

      ?
      if [ -f /opt/local/etc/bash_completion ]; then
          . /opt/local/etc/bash_completion
      fi

      You can now use bash completion, i.e. press tab to auto-complete branch names, git commands etc. for you. E.g. to checkout a branch named ecomm-backoffice-3.31.x, just type this:

      ?
      # git ch<tab> ec<tab>b<tab>3.31.x

      Double pressing tab to get suggestions also works as expected.

      Using the OS Keychain

      To not have to enter your password all the time, also add this to your ~/.gitconfig

      ?
      [credential]
              helper = osxkeychain

      UI

      Sometimes you might want to use a UI to do certain things in git. A good example would be to look at the commit history, especially how branches are related etc. The good thing is, git comes with a cross platform UI to handle this. It might not look pretty (it's written in TK), but it is very functional and you can take it with you when you switch platforms. I first used it under Linux when I first learned git, then I had to use a Windows box at some other company and now at NZone I am using a Mac. Same UI, nothing new to learn. I'm aware that many people will want to have something "pretty" or integrated with their IDE. If you find something that appeals to you, please add it here so we can all benefit from it, I'm just not that person.

      gitk

      THE definitive tool to display version history and search through commits (messages, contents, paths, whatever). By default, gitk will only display the branch that you are currently on. If you want to display all the branches your repository knows about, use the following command:

      ?
      # gitk --all

      I always configure an alias in my bash for this, so I don't have to type so much:

      ?
      alias gitka='gitk --all'

      Put this in your .bash_profile.

      gitgui

      There's also gitgui, which personally, I really don't use, so I can't say anything about usability or quirks.

      Eclipse

      Since Kepler, the eclipse plugin for git is part of the eclipse distribution. Personally, I disabled it and still use the command line, as I use git with other projects, that are non-java too, so my toolbox knowledge applies regardless of what I am putting into git.

      Netbeans

      The Netbeans IDE also supports git, but I personally don't use Netbeans, so I can't say anything about it.

      SourceTree

      SourceTree is a Git client application to manage all your repositories, hosted or local, through a simple and intuitive visual interface.

      You can download it for free in the following link if you wish to say goodbye to the command line: SourceTreeApp

      Advanced topics

      Here's the ECOMM specific version of bisection left, as the ECOMM part has been removed when moving this page over to Build Management, but it might be interesting to see how it worked

      Bisection

      SVN: not possible

      Sometimes you may have a bug in your software, but you don't know which commit introduced the bug. If you have a way to reproduce the bug reliably, git can help you find the offending commit. If you can script the reproduction of the bug, you can even fully automate it. What bisection does, is basically just a binary search through history. You give it a known good commit and a known bad commit. Git will then automatically start checking out commits in between the two and ask you to tell it, whether that commit was a good one or a bad one. This approach was used to find the commit for the following bug for example:  https://jirapd.corp-apps.com/browse/ECOMM-1553.

      ?
      # git bisect start master 5af6a8f

      This will start bisection, telling git that the commit master(which can be a branch head or tag or a commit hash) is bad, i.e. has a defect and 5af6a8f, which is a commit hash in this case, is the last known good commit, that doesn't have the defect. You now have to reproduce the defect and tell git, whether you could or not. Let's say you couldn't:

      ?
      # git bisect good

      Git will now check out another revision and you repeat the whole thing. Let's say this time the defect could be reproduced, so you do this:

      ?
      # git bisect bad

      If there are still some commits left in between the good one and the bad one, this will continue in a cycle, until you have found the border between good and bad. You can also tell git to skip a commit, e.g. because it chose a commit that doesn't compile or has some other known bug, that prevents you from reproducing the problem with that commit. It will then simply check out an adjacent commit for you to try:

      ?
      # git bisect skip

      This is the manual process. But lets say you have 200 commits to go through, which will take quite a while even with a binary search or building and reproducing the bug takes 2 hours each time. If you can scripts the building, deploying and reproduction of the bug and decide whether the defect is present, you can let git run the bisection completely automated:

      ?
      # git bisect run reproduce-ECOMM-1553.sh

      This assumes that there is a script called reproduce-ECOMM-1553.sh, which exits with either a return code of 0, 1 or 125. 125 is like git bisect skip, while 0 and 1 are for good and bad respectively. See http://lwn.net/Articles/317154/ for more examples or if you think that this is too much to be bothered with.
