      Git rebase

      Skip to end of metadata
      • Created by Alex G - Apical Services, last modified on 20-May-2015
      Go to start of metadata

       

       

      Remember how the Git Howto told you not to use rebase? Well I use it all the time, and I actually wrote that How To.

      So how come I said not to use it? CYA, really. I didn't want anybody to blame me if something goes wrong with your rebase. Well here you go: Use it, it's awesome!

      I actually have been using the rebase feature every single time I made a change during the last two years, because I was using git svn and it requires git to straighten out the commit history for being able to commit to SVN.

      SVN version history

      SVN has a straight version history, unlike git, where things can happen in parallel. In SVN, everyone has to "svn update" before doing an "svn commit" and commits go straight to the server. In such a situation, your trunk version history will always be in a straight line, as every commit is directly based upon the last commit in the repository. So in SVN, history for trunk looks like this basically (the dots are meant to be commits):

      Git version history

      That's different in git, because you actually commit locally and there are lots of different version histories "happening" on every developer's computer. When it's time to push, git doesn't require you to "straighten out" history. It will happily just connect all the different little trees that go around on everyone's computer with a "merge commit".

      Let's supposed two developers were working on the same code base in git, then version history might look like this:

      Developer 1 started working on hist changes locally based upon the first commit shown in the graph, maybe because he forgot to do a pull before starting. Developer 2 however, started from the second commit (that "someone" pushed to the repository, maybe developer 3 did that). Both work and make their own commits and then at some point push to the repository. Meanwhile unfortunately developers 3 pushed another commit to the repository and developer 2 has to do a merge commit now. Git handles this basically automatically for you, unless there are conflicts, so you may not even have noticed. Go check the history of the ecomm-api project in some sort of graphical viewer to see what I mean. The graph shouldn't look too unlike the above image. Now developer 1 wants to push his commits and has to do the exact same thing, because developers 2 and 3 pushed commits. If you have lots of developers working on the same project, this can get really messy looking.

      Git rebase

      So what's the purpose of git rebase?

      Git rebase let's you change history!

      Looking back at the graph from before, let's say that developer 1 wants to "straighten out" the way that the commit history graph looks. Before he pushes any of his commits, he will use git rebase, to make it look like he actually started working on his stuff only after developer 2 already pushed his commits to the repository!

      It's really easy. All you have to do is to git rebase origin/master. That's the same on everyone's computer, as by default "origin" is the name of the repository. So "origin/master" refers to the master branch on the remote repository, while your local tracking branch for that is just called master:

      ?
      # git rebase origin/master
      First, rewinding head to replay your work on top of it...
      Applying: Commit message blah blub

      This will make history look like this (globally speaking):

      This doesn't actually look much better you say. Ah but that's just because of the way I drew it to make you see which developer did which commits. In the actual version history in some sort of graphical git tool, it would look more like this:

      Without looking at individual commits and who made them and when, there's no way to tell that developer 2 actually did the last two commits shown. There really is no way to tell that he started out working from the very first commit shown in the image and that he rebased on top of the commit from developer 2. However, the "branching off" that developer 2 did, is actually represented in the graph visually and can be seen even if you just look at "git log" output, because the third commit from the bottom will show up with a commit message of "Merge commit..." and look like it's "empty", i.e. no realy code changes made.

      In a hurry?

      As suggested below by Alberto:

      ?
      # git pull --rebase

      will work just fine as well, basically doing what git svn has been doing for me all the time.

      So why did you really say not to use it?

      Well really just because we all weren't used to using git and it can be a bit frightening, if you rebase and you get conflicts and you're afraid you might be loosing all your work and/or you do something to actually loose your changes. As with everything: Just practice it. Maybe on a small commit that took like 5 minutes to code up.

      2. Alberto Manzaneque

        One thing that I always do is 'pull --rebase'.

        When you have a local commit that you want to push, but there are other updates in the repository, you are forced to 'pull' first. But if you just run 'git pull', git cosiders your local commit as a different branch and creates a merge commit to merge your "fake" branch and the change in the repo.

        But if you run 'git pull --rebase', git pulls the content from the repo and then replays your commit on top of it, resulting in a linear svn-like time line.
