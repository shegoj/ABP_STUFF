      One git clone is enough!

      Skip to end of metadata
      • Created by Alex G - Apical Services on 19-May-2015
      Go to start of metadata

       

      Today I want to shine some more light onto a topic that may be a bit daunting if you've used SVN for all your life. Working with branches.

      The git guide already covers it a bit How-To: Working with git - switching branches but I'd like to point out some of the workflows that you may encounter in real life and how they can elegantly work in git.

      The SVN way

      You need to be online first and checking out a branch can take a while on a slow network, as it basically re-downloads the whole project again. In SVN you have to check out a branch in a separate directory. You then have several options, none of which sounds great. Import that into eclipse, making sure that you name things appropriately, so the project names don't clash. Or use a different eclipse workspace for the branches and switch eclipse workspaces all the time. Or you rename/copy folders around. Needless to say that this is a major hassle and nobody likes working with branches and it's easy to make mistakes.

      The git way

      In git, all of this goes away.

      One folder

      You only need exactly one folder in git and you just keep switching branches within that folder, whenever you need to. This means that you only need to clone once. So even if, theoretically, you had 5GB of data in the repository and had to download it over a slow line over the Atlantic, that's not the end of the world. One developer on the "wrong side of the Atlantic" can download the 5GB and then just share their ".git" directory locally with all other developers. That said, my ecomm library ".git" directory is all of 185MB big.

      Changing branches is a very fast operation in git, since you have a local copy of all branches already anyway and because most branches aren't all that different from master or other branches and git will just replace the few files that need replacing in your working copy. Example:

      Master is on 3.57-SNAPSHOT:

      ?
      ecomm$ time git checkout ecomm-3.56.x
      Switched to branch 'ecomm-3.56.x'
      real    0m0.104s

      So it took all of 100ms to switch to the branch for our current LINT version to make some change there. This will be the most common cause for you to switch branches probably. Now let's try a really old branch, just to see what happens:

      ?
      ecomm$ time git checkout ecomm-3.30.x
      Switched to branch 'ecomm-3.30.x'
      real    0m0.989s

      It took all of a whole second to switch to code that is 26 sprints old, or 78 weeks or about 1.5 years.

      But this doesn't work with eclipse!

      I know, "this can't work". When I first switched to git I may have thought the same. It works beautifully though. You just have to accept that you need to change the way you work a bit and once you do, you will notice that it's better! Just learn your tool and take advantage of it. If you try to work against it, it will be a pain, you will be miserable and you will hate it. That's what people said when SVN came along and replaced CVS.

      All you need in eclipse, is exactly one workspace with all your projects in it and you just switch to the branches you need to work with "right now", whenever you need to. So your eclipse can look like this (I use eclipse "Working sets", one per project, as maven module naming will mix them all up otherwise):

      Depending on whether you use the eclipse plugin for git or not, you may need to refresh the projects that you switched to another branch. If you use the command line, like me, you do have to refresh the whole project, which is where the working sets come in, so that you can easily refresh everything belonging to the one or two projects that you switched, without having to wait for a refresh on everything on your workspace.

      Note that when you switch a project to another branch, the whole project will be switched. You can't check out just a sub folder and work on that, like you did in SVN. Yes I'm looking at you, I-Shaped QAA. The following image shows how to do it, if you are using the eclipse plugin:

      And that's really all there is to it.

      Note: The following may or may not be supported in the eclipse git plugin version you use and may or may not be supported in the other graphical tool you are using. The eclipse git plugin is notorious for renaming stuff from their real names too. So the following is described in terms of the command line. You may or may not be able to translate those workflows to whatever graphical tool you are using. I heard Yiang Guo has a nice Mac tool, so he may be able to help you with trying this out.

      But I already started working on regression week stuff!

      As long as we still have regression week, you may already have started working on regression week tasks, when someone finds a bug and you need to fix it on the LINT branch(es). Your first thought is to clone all the projects again and switch them to the right branch and ... yeah, don't!

      There are two scenarios here.

      1. You already committed whatever you were working on in master
      2. or more likely you are in the middle of something and have lots of uncommitted changes, that may not even compile right now and that you can't commit and switch branches.

      In the first case, perfect, just switch branches, nothing to worry about. In the second case, use stash! How-To: Working with Git Working with Git-Stash.

      So here's the scenario. I've been refactoring some of our aspects (the ... means looots more files changed) and this totally doesn't even compile right now and someone needs a bug fixed for LINT and it's Friday 2 o'clock.

      ?
      ecomm$ git status
      On branch master
      Changes not staged for commit:
        (use "git add <file>..." to update what will be committed)
        (use "git checkout -- <file>..." to discard changes in working directory)
          modified:   ecomm-common/src/main/java/ecomm/common/logging/LoggingAspect.java
      ....
      Untracked files:
        (use "git add <file>..." to include in what will be committed)
          ecomm-common/src/main/java/ecomm/common/security/SecurityAspect.java
      no changes added to commit (use "git add" and/or "git commit -a")

      I can temporarily stash away my changes in a safe place, that git handles for me automatically and switch to the branch.

      Note: Stash only stashes away changes to files that git "knows about". Technically it doesn't "know" about files that are new. That meanas that "SecurityAspect.java" would not be stashed. That might be OK and exactly what I wanted in a different scenario, but here, I want it stashed away as well, because that file should definitely not go onto the LINT branch. You can also stash as many times as you want and in as many branches and places as you like.

      ?
      ecomm$ git add ecomm-common/src/main/java/ecomm/common/security/SecurityAspect.java
      ecomm$ git stash
      Saved working directory and index state WIP on master: d51e1ff ECOMM-6158 - Retrieve balance from intapi - Account info integration with intapi - Unified intapi integration services
      HEAD is now at d51e1ff ECOMM-6158 - Retrieve balance from intapi
      ecomm$ git checkout ecomm-3.56.x
      Switched to branch 'ecomm-3.56.x'

      Now I can do my fix, test it, commit it, push it etc. and once I'm done, I just go back to finishing up the work I did for regression week:

      ?
      ecomm$ git stash pop
      On branch ecomm-3.56.xOn branch ecomm-3.56.x
      Changes to be committed:
        (use "git reset HEAD <file>..." to unstage)
          new file:   ecomm-common/src/main/java/ecomm/common/security/SecurityAspect.java
      Changes not staged for commit:
        (use "git add <file>..." to update what will be committed)
        (use "git checkout -- <file>..." to discard changes in working directory)
          modified:   ecomm-common/src/main/java/ecomm/common/logging/LoggingAspect.java
      no changes added to commit (use "git add" and/or "git commit -a")
      Dropped refs/stash@{0} (7ea6aa7ce33f10ce420a0d478045b7928ee856eb)

      There we go, safely brought back all my changes. Note that because I already added the new file to stash it, git now "knows" about it and will commit it, once I am ready to commit.

      If you are still reading: What did I do wrong and will "break my neck" now?

      I accidentally started making changes for my regression task on the LINT branch!

      Right, I forgot to switch back to master, before getting my changes back from the stash. This happens. It's not a big deal. It doesn't only happen if you had to stash, it also happens if you've made a change on a branch, then went home for the day and forget you didn't switch back to master before starting work in the morning and in a myriad of other situations.

      The easy fix: Just switch to master!

      ?
      ecomm$ git status
      On branch ecomm-3.56.x
      Changes to be committed:
        (use "git reset HEAD <file>..." to unstage)
          new file:   ecomm-common/src/main/java/ecomm/common/security/SecurityAspect.java
      Changes not staged for commit:
        (use "git add <file>..." to update what will be committed)
        (use "git checkout -- <file>..." to discard changes in working directory)
          modified:   ecomm-common/src/main/java/ecomm/common/logging/LoggingAspect.java
      ecomm$ git checkout master
      M    ecomm-common/src/main/java/ecomm/common/logging/LoggingAspect.java
      A    ecomm-common/src/main/java/ecomm/common/security/SecurityAspect.java
      Switched to branch 'master'
      ecomm$ git status
      On branch master
      Changes to be committed:
        (use "git reset HEAD <file>..." to unstage)
          new file:   ecomm-common/src/main/java/ecomm/common/security/SecurityAspect.java
      Changes not staged for commit:
        (use "git add <file>..." to update what will be committed)
        (use "git checkout -- <file>..." to discard changes in working directory)
          modified:   ecomm-common/src/main/java/ecomm/common/logging/LoggingAspect.java

      Here we are, git switched the branch back to master and kept all the modifications that weren't committed yet. That also works if there's a file that git doesn't "know about" yet.

      But I committed to the LINT branch before I noticed!

      ?
      ecomm$ git commit -a -m 'blah'
      [ecomm-3.56.x 16731c1] blah
       2 files changed, 68 insertions(+), 1 deletion(-)
       create mode 100644 ecomm-common/src/main/java/ecomm/common/security/SecurityAspect.java

      No problem (unless you also pushed). Since commits are made locally in git, you can fix ANY situation you get yourself into, unless you have already pushed to the repository. Even then you can "fix" things, but it might look weird in the history.

      A possibility here would be to just cherry-pick the commit over to the master branch. All we need is the commit hash of the "wrong commit" from the LINT branch, which it conveniently lists right after committing: 16731c1

      ?
      ecomm$ git checkout master
      Switched to branch 'master'
      KOLM10-AGRETENC:ecomm agretencord$ git cherry-pick 16731c1
      [master a230386] blah
       2 files changed, 68 insertions(+), 1 deletion(-)
       create mode 100644 ecomm-common/src/main/java/ecomm/common/logging/LoggingAspect2.java

      Now you need to "clean up" the LINT branch, as it still has the commit that you don't want to have there. There's a neat little syntax in git, that lets you "navigate" in commit without knowing the actual commit hashes. We already know the commit hash of the "wrong commit", it's 16731c1. Using the caret character "^", we can tell git that we want "the commit before that": "16731c1^".

      ?
      ecomm$ git checkout 16731c1^
      Note: checking out '16731c1^'.
      You are in 'detached HEAD' state. You can look around, make experimental
      changes and commit them, and you can discard any commits you make in this
      state without impacting any branches by performing another checkout.
      If you want to create a new branch to retain commits you create, you may
      do so (now or later) by using -b with the checkout command again. Example:
        git checkout -b new_branch_name
      HEAD is now at 02f4d74... ECOMM-6486: Rename BO configuration item to "One-click deposit channels"

      Note the fat message that we're in "detached HEAD" state. This is nothing to worry about, as we're going to get ourselves out of that situation real soon. It basically means that if you start making changes to the code in this situation, you may loose those changes if you aren't careful. But even if you do start making changes, that's nothing to worry about, as long as you know how to get out of that situation. But that's another story. So back to fixing up the LINT branch. All we need to do is to actually delete the branch. That's fine, because you really only delete the local branch on your computer, which has "nothing" to do with the branch on the repository. After that, you really don't have to do anything, if you don't want to keep making changes to the LINT branch. If you do, you need to synchronize your local branch label with the branch label on the repository:

      ?
      ecomm$ git branch -D ecomm-3.56.x
      Deleted branch ecomm-3.56.x (was 16731c1).
      ecomm$ git checkout ecomm-3.56.x
      Previous HEAD position was 02f4d74... ECOMM-6486: Rename BO configuration item to "One-click deposit channels"
      HEAD is now at 9bbeb42... BLD-63686:prepare for next development iteration

      Your graphical tool of choice may even offer the above as a much simpler feature, something like "move branch label" or something like that.

      Remember: "Everything" in git is just labels for commits. You can relabel stuff all you like and play around locally. Actually, to be able to paste all these code blocks, I actually DID the stuff I'm showing on my real ecomm library project. Now that I'm done, I'm just gonna get rid of the commits I made, switch to master and reset any changes that may be there.

        1. Alex G - Apical Services AUTHOR

          Yes. That resets the branch pointer back as well, but will leave any new files you had added in your commit in your working tree for you to delete and any modifications that you made will still be present. Doing a "git reset --hard HEAD^" would fix that though (i.e. get rid of those changes as well). Basically it's a faster way to do it, if you also need to have the pointer re-attached (which you probably want on master, but probably don't need to on a branch like in the example, where you'd leave out the last checkout step anyway).
