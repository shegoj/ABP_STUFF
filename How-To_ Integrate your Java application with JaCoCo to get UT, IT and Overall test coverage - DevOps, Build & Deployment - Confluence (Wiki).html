      How-To: Integrate your Java application with JaCoCo to get UT, IT and Overall test coverage

      What is JaCoCo?

      According to their official website, JaCoCo is a free code coverage library for Java, which has been created by the EclEmma team based on the lessons learned from using and integration existing libraries for many years.

      It provides the following features:

      • Coverage analysis of instructions (C0), branches (C1), lines, methods, types and cyclomatic complexity.
      • Based on Java byte code and therefore works also without source files.
      • Simple integration through Java agent based on-the-fly instrumentation. Other integration scenarios like custom class loaders are possible through the API.
      • Framework agnostic: Smoothly integrates with Java VM based applications like plain Java programs, OSGi frameworks, web containers or EJB servers.
      • Compatible with all released Java class file versions.
      • Support for different JVM languages.
      • Several report formats (HTML, XML, CSV).
      • Remote protocol and JMX control to request execution data dumps from the coverage agent at any point in time.
      • Ant tasks to collect and manage execution data and create structured coverage reports.
      • Maven plug-in to collect coverage information and create reports in Maven builds.

      Requirements

      1. Java project. JaCoCo is designed for Java applications only (Java Code Coverage). If you are a PHP/NodeJS/Javascript developer, we will need to look for a different solution for you Feel free to ask #Build-Management about it.
      2. Parent POM 1.4.1. In this version we have changed the Maven plugin that will run your integrated tests, from maven-surefire-plugin, to maven-failsafe-plugin. This is to avoid global failures in case a test doesn't success.
      3. Use of Maven >= 3.0. Requirement to use Parent POM >= 1.1
      4. BGT framework. In order to run integration tests, your software project must be already integrated with BGT. Old projects using JQA will not be supported.
      5. Bamboo plans. The whole process will be triggered through Bamboo, using the Stats plan. Once completed, report will be uploaded to Sonar

      Implementation


      [Unit Test coverage] Step-by-step guide

      1. Declare JaCoCo Maven plugin in project's main pom.xml file

      <PROJECT>/pom.xml
      ?
      ...
      <build>
        ...
        <pluginManagement>
          ...
          <plugin>
            <groupId>org.jacoco</groupId>
            <artifactId>jacoco-maven-plugin</artifactId>
            <version>${jacoco.version}</version>
          </plugin>
          ...
        </pluginManagement>
        ...
      </build>
      ...
      <properties>
        <jacoco.version>0.7.4.201502262128</jacoco.version> <!-- 0.7.5 is not supported yet due to a bug in Sonar -->
      </properties>
      ...

       

      2. Define two JaCoCo executions. One to launch the agent, and another to generate reports in HTML, XML and CSV (This may be useful for developers to know the quality of their tests before committing changes).

      <PROJECT>/pom.xml
      ?
      <build>
        ...
        <plugins>
          <plugin>
            <groupId>org.jacoco</groupId>
            <artifactId>jacoco-maven-plugin</artifactId>
            <executions>
              <execution>
                <id>agent-for-ut</id>
                <goals>
                  <goal>prepare-agent</goal> <!-- This goal is bound by default to 'initialize' phase. -->
                </goals>
                <configuration>
                  <!-- When working on multi-module projects, by default one report file is generated per module.
                       The following two parameters are intended to avoid this behavior. 
                       It's easier to work with one single results file.
                  -->
                  <destFile>${jacoco.ut.binary.report.file}</destFile>
                  <append>true</append>
                </configuration>
              </execution>
              <execution>
                <id>report-ut</id>
                <goals>
                  <goal>report</goal> <!-- Bound by default to 'verify' phase -->
                </goals>
                <configuration>
                  <!-- Let's tell JaCoCo where to look for the report file (binary), and output folder for
                       XML, HTML and CSV files generation 
                  -->
                  <dataFile>${jacoco.ut.binary.report.file}</dataFile>
                  <outputDirectory>${jacoco.ut.reports.path}</outputDirectory>
                </configuration>
              </execution>
            </executions>
          </plugin>
          ...
        </plugins>
      </build>
      ...
      <properties>
        ...
        <jacoco.ut.reports.path>${project.basedir}/../target/jacoco-ut-reports/${project.artifactId}</jacoco.ut.reports.path>
        <jacoco.ut.binary.report.file>${project.basedir}/../target/jacoco.exec</jacoco.ut.binary.report.file>
        ...
      </properties>
      ...

       

      3. Test the integration locally

      Now, run 'mvn clean verify' from your local computer. During the execution you must see a system process similar to:

      ps ax | grep jacoco
      ?
      24188 pts/0    Sl+    0:27 /usr/java/jdk1.8.0_45/jre/bin/java -javaagent:/maven2/org/jacoco/org.jacoco.agent/0.7.4.201502262128/org.jacoco.agent-0.7.4.201502262128-runtime.jar=destfile=/home/bamboo/sandbox/casino-integration-services/casino-integration-services-webapp/../target/jacoco.exec,append=true -jar /home/bamboo/sandbox/casino-integration-services/casino-integration-services-webapp/target/surefire/surefirebooter6891113968231967373.jar /home/bamboo/sandbox/casino-integration-services/casino-integration-services-webapp/target/surefire/surefire5319318361201733026tmp /home/bamboo/sandbox/casino-integration-services/casino-integration-services-webapp/target/surefire/surefire_62988606218771322323tmp

       

      Once the execution is finished, the following elements must be present in $PROJECT/target folder:

      ls -lh target/
      ?
      total 780K
      -rw-rw-r--  1 bamboo bamboo 771K Jul 17 05:22 jacoco.exec
      drwxrwxr-x 10 bamboo bamboo 4.0K Jul 17 05:22 jacoco-ut-reports

       

      4. Configure Sonar to process UT results file

      <PROJECT>/pom.xml
      ?
      <properties>
        <!-- Sonar -->
        <sonar.language>java</sonar.language>
        <sonar.dynamicAnalysis>reuseReports</sonar.dynamicAnalysis>
        <sonar.java.coveragePlugin>jacoco</sonar.java.coveragePlugin>
        <sonar.jacoco.reportPath>${jacoco.ut.binary.report.file}</sonar.jacoco.reportPath>
      </properties>

       

      5. Launch Sonar plan from Bamboo

      To get code coverage, it is mandatory to run tests as part of Sonar execution. You might need to change Maven's goal in your Stats plan from "sonar:sonar" to "clean test sonar:sonar".

      Go to your Stats plan in Bamboo, and launch it! Once completed, browse Sonar (Or look for the direct link, which can be found in execution's log), and look for 'Unit test coverage' widget. It should display values different than 0% (Speak with your developers otherwise ).

      That's it!

      [Integration Test coverage] Step-by-step guide

      This is quite a long process. First of all, let's enumerate all elements involved on it:

      1. JaCoCo agent running on application side (As Java agent) in MDEV (And maybe LINT). Agent will be working in "tcpserver" mode, to allow connections from Bamboo servers, for data collection purposes.
      2. Maven project. Almost all work for this step will be done in <$PROJECT>-tests sub-module (Which must exist since it's a requirement for BGT). Also, every plugin declaration/configuration will go under a profile NOT active by default. We will be using maven-antrun-plugin and maven-enforcer-plugin. Ironically, jacoco-maven-plugin does not take part of this process, since it has some lacks (i.e. Is unable to tell JaCoCo agent to reset test counters) and that step (And the one to obtain the IT results file from remote server) are controlled with some Ant tasks. However, to achieve step #1, we will need to modify certain files located in other modules (Assembly file, custom.sh, project's properties file, etc).
      3. Product-WS. Our preferred configuration management service!
      4. Sonar. In the same way as with unit tests coverage, this JaCoCo report will be also uploaded to Sonar. Just one clarification about this step: When you finish this chapter, in order to get both UT and IT results in your Sonar report, you will need to launch a full regression as part of the Stats plan.

       

      1. Run JaCoCo agent in a Java project on server-side

      Required modifications to Maven project. For this example, I have taken "Casino Integration Services" as example, which uses Tomcat-Overlay.

      First, define the agent in project's root POM file.

      $PROJECT/pom.xml
      ?
      <project>
        ...
        <dependencyManagement>
          <dependencies>
            ...
            <!-- JaCoCo agent -->
            <dependency>
              <groupId>org.jacoco</groupId>
              <artifactId>org.jacoco.agent</artifactId>
              <version>${jacoco.version}</version> <!-- Let's try and use the same version of JaCoCo for UT and IT -->
            </dependency>
            ...
          </dependencies>
        </dependencyManagement>
        ...
      </project>

       

      Define it in $PROJECT-site POM and assembly files to make it available on application startup.

      $PROJECT/$PROJECT-site/pom.xml
      ?
      <project>
        ...
        <dependencies>
          ...
          <dependency>
            <groupId>org.jacoco</groupId>
            <artifactId>org.jacoco.agent</artifactId>
          </dependency>
          ...
        </dependencies>
        <build>
          <plugins>
            <plugin>
              <groupId>org.apache.maven.plugins</groupId>
              <artifactId>maven-dependency-plugin</artifactId>
              <executions>
                <execution>
                  <id>dependency:unpack</id>
                  <phase>generate-resources</phase>
                  <goals>
                    <goal>unpack</goal>
                  </goals>
                </execution>
              </executions>
              <configuration>
                <artifactItems>
                  ...
                  <artifactItem>
                    <groupId>org.jacoco</groupId>
                    <artifactId>org.jacoco.agent</artifactId>
                    <outputDirectory>${project.build.directory}/jacoco/</outputDirectory>
                    <includes>jacocoagent.jar</includes>
                  </artifactItem>
                </artifactItems>
              </configuration>
            </plugin>
            ...
          </plugins>
        </build>  
        ...
      </project>
      Assembly file
      ?
      <?xml version="1.0" encoding="UTF-8"?>
      <assembly xmlns="http://maven.apache.org/xsd/assembly-1.1.2.xsd">
        <id>tomcat</id>
        <formats>
          <format>tar.gz</format>
        </formats>
        <baseDirectory>${server.name}</baseDirectory>
        <includeBaseDirectory>true</includeBaseDirectory>
        <fileSets>
          ...
          <fileSet>
            <directory>${project.build.directory}/jacoco</directory>
            <outputDirectory>lib/</outputDirectory>
            <includes>
              <include>jacocoagent.jar</include>
            </includes>
          </fileSet>
          ...
        </fileSets>
        ...
      </assembly>

       

      Then, create the profile for JaCoCo in your $PROJECT-tests module.

      $PROJECT/$PROJECT-tests/pom.xml
      ?
      <project xmlns="http://maven.apache.org/POM/4.0.0" xmlns:xsi="http://www.w3.org/2001/XMLSchema-instance" xsi:schemaLocation="http://maven.apache.org/POM/4.0.0 http://maven.apache.org/xsd/maven-4.0.0.xsd">
        ...
        <!-- QA Automation Profiles -->
        <profiles>
          <profile>
            <id>jacoco</id>
            <properties>
              <!-- 
              The following property must be passed to Maven execution to know where JaCoCo is listening
              It should be setup by adding -Djacoco.address=<HOSTNAME> in command line (Or Bamboo plan).
              If not passed, build will fail (Check maven-enforcer-plugin below)
              -->
              <jacoco.remote.address>${jacoco.address}</jacoco.remote.address>
              <!-- 
              JaCoCo default port when running in 'tcpserver' mode. 
              Its modification is not recommended... You could face ACL issues  
              -->
              <jacoco.remote.port>6300</jacoco.remote.port>
              <!-- Tells JaCoCo where to put results file -->
              <jacoco.it.report.file>../target/jacoco-it.exec</jacoco.it.report.file>
            </properties>
            <build>
              <pluginManagement>
                <plugins>
                  <plugin>
                    <groupId>org.apache.maven.plugins</groupId>
                    <artifactId>maven-antrun-plugin</artifactId>
                    <version>1.8</version>
                  </plugin>
                </plugins>
              </pluginManagement>
              <plugins>
                <plugin>
                  <groupId>org.apache.maven.plugins</groupId>
                  <artifactId>maven-enforcer-plugin</artifactId>
                  <executions>
                    <execution>
                      <id>enforce-property</id>
                      <goals>
                        <goal>enforce</goal>
                      </goals>
                      <configuration>
                        <rules>
                          <!-- This rule will produce a build failure if ${jacoco.address} is not set -->
                          <requireProperty>
                            <property>jacoco.address</property>
                            <message>You must pass -Djacoco.address=$HOSTNAME to this Maven execution!</message>
                          </requireProperty>
                        </rules>
                        <fail>true</fail>
                      </configuration>
                    </execution>
                  </executions>
                </plugin>
                <plugin>
                  <groupId>org.apache.maven.plugins</groupId>
                  <artifactId>maven-antrun-plugin</artifactId>
                  <executions>
                    <!-- This execution resets JaCoCo counters. Necessary to produce real data. -->
                    <execution>
                      <id>reset-jacoco-counters</id>
                      <phase>pre-integration-test</phase>
                      <configuration>
                        <target xmlns:jacoco="antlib:org.jacoco.ant">
                          <taskdef uri="antlib:org.jacoco.ant" resource="org/jacoco/ant/antlib.xml" classpathref="maven.test.classpath" />
                          <jacoco:dump address="${jacoco.remote.address}" port="${jacoco.remote.port}" reset="true" destfile="${jacoco.it.report.file}"                  append="false"/>
                          <delete file="${jacoco.it.report.file}"/>
                        </target>
                      </configuration>
                      <goals>
                        <goal>run</goal>
                      </goals>
                    </execution>
                    <!-- 
                    Second Ant execution. It will attack ${jacoco.address} to request the dumpfile with test results. 
                    Dump file (jacoco-it.exec) will be placed in project's root target folder.
                    -->
                    <execution>
                      <id>get-jacoco-report</id>
                      <phase>post-integration-test</phase>
                      <configuration>
                        <target xmlns:jacoco="antlib:org.jacoco.ant">
                          <taskdef uri="antlib:org.jacoco.ant" resource="org/jacoco/ant/antlib.xml" classpathref="maven.test.classpath" />
                          <jacoco:dump address="${jacoco.remote.address}" port="${jacoco.remote.port}" dump="true" reset="true" destfile="${jacoco.it.report.file}" append="false"/>
                        </target>
                      </configuration>
                      <goals>
                        <goal>run</goal>
                      </goals>
                    </execution>                       
                  </executions>
                  <dependencies>
                    <dependency>
                      <groupId>org.jacoco</groupId>
                      <artifactId>org.jacoco.ant</artifactId>
                      <version>${jacoco.version}</version>
                    </dependency>
                  </dependencies>
                </plugin>
              </plugins>
            </build>
          </profile>
          ...
        </profiles>
        ...
      </project>

       

      2. Control JaCoCo execution on application startup.  

      Create a logic to include JaCoCo agent in $JAVA_OPTS, depending on the value of one property. Note that the code below might be slightly different, depending on each project, overlay used, etc.

      custom.sh (Usually located in $PROJECT/$PROJECT-site/src/main/resources/scripts/)
      ?
      #!/bin/bash
       
      # JaCoCo
      JACOCO_OPTS=""
      if [ ${casino_casinointegrationservices_jacoco_enabled} -eq 1 ] ; then
          JACOCO_OPTS="-javaagent:${INSTALL_BASE}/lib/jacocoagent.jar=destfile=${INSTALL_BASE}/jacoco-it.exec,output=tcpserver,address=*"
      fi
      export JAVA_OPTS="${JACOCO_OPTS} $JAVA_OPTS"

       

      Include the property defined in custom.sh file, defined above.

      Project's properties file
      ?
      # JaCoCo
      casino.casinointegrationservices.jacoco.enabled=@@casino.casinointegrationservices.jacoco.enabled@@

       

      Follow How-To: Create a New Key or Key Override and How-To: Add Configuration to a Deployable to add such key to your application's configuration group. Key default value must be '0', with override of '1' in MDEV. We are working in a solution for the rest of environments, which won't require any extra effort from your side.

       

      3. Deploy your application in MDEV to check if JaCoCo agent is running as well.

      Once above steps are completed, use your Bamboo plan to deploy your application. You can verify that JaCoCo has started by checking the Java process in the server:

      ps ax | grep java | grep jacoco
      ?
      11603 ?        Sl     4:16 /usr/local/java/bin/java ... -javaagent:/sites/casino-integration-services.brand--x.com/lib/jacocoagent.jar=destfile=/sites/casino-integration-services.brand--x.com/jacoco-it.exec,output=tcpserver,address=* 
      -Dtomcat.host=0.0.0.0 -Dtomcat.port=8080 -Dtomcat.port.ssl=8443 -Dtomcat.port.ajp=8009 -Dtomcat.port.shutdown=8005 
      -Dtomcat.maxHttpHeaderSize=8192 -Dtomcat.maxThreads=300 ... org.apache.catalina.startup.Bootstrap start

       

      You can also check if connection to port 6300 is opened:

      telnet casino-integration-services-1.mdev.brand--x.com 6300
      ?

      Escape character is '^]'.

      ALL DONE! Your application is now happily integrated with JaCoCo. It will be logging any event at application level, waiting to serve the report upon request.


      4. Test the integration locally

      To test this, a Maven execution with 'automation' and 'jacoco' profiles enabled is required. You should be able to run the following command from your local computer.

      ?
      #> mvn clean verify -Pautomation,jacoco -Djacoco.address=$TARGET_HOST

      This command will require:

      • JaCoCo agent running on application side, in 'tcpserver' mode.
      • Property "jacoco.address" pointing to target server without http(s)://.

      This command will execute:

      • A clean execution of Maven (Every target folder will be removed to ensure 'fresh' results)
      • Default's Maven life-cycle until 'verify' goal. Installation to local repository and deploy to Nexus will be skipped (This is achieved via the Publish plan) 
      • A FULL suite of integration tests will be run, as part of 'automation' profile.
      • All plugins and executions described in 'jacoco' profile.

      This command will produce:

      • $PROJECT/target/jacoco.exec (Unit tests binary report) *
      • $PROJECT/target/jacoco-it.exec (Integration tests binary report)
      • $PROJECT/target/jacoco-ut-reports/ (HTML, XML and CSV reports) * **
      • Other reports not directly related to JaCoCo (failsafe-reports, surefire-reports, etc). **

      * Only if you implemented [Unit test coverage] chapter!

      ** You might find useful to capture these reports using Artifact definitions in Bamboo

       

      - "So... how do I know if my Maven execution has reset JaCoCo counters?"

      You should see a message similar to this:

      ?
      [INFO] --- maven-antrun-plugin:1.8:run (reset-jacoco-counters) @ casino-integration-services-tests ---
      [INFO] Executing tasks
       
      main:
      [jacoco:dump] Connecting to casino-integration-services-1.mdev.brand--x.com/10.102.91.81:6300
      [jacoco:dump] Dumping execution data to /bamboo/xml-data/build-dir/BRNGCASINO-CASINOINTEGRATIONSERVICESSTATS-JOB1/target/jacoco-it.exec
      [delete] Deleting: /bamboo/xml-data/build-dir/BRNGCASINO-CASINOINTEGRATIONSERVICESSTATS-JOB1/target/jacoco-it.exec

      - "And once integration tests are finished, how can I be sure that Maven has requested test results?"

      Message should be like this:

      ?
      [INFO] --- maven-antrun-plugin:1.8:run (get-jacoco-report) @ casino-integration-services-tests ---
      [INFO] Executing tasks
       
      main:
      [jacoco:dump] Connecting to casino-integration-services-1.mdev.brand--x.com/10.102.91.81:6300
      [jacoco:dump] Dumping execution data to /bamboo/xml-data/build-dir/BRNGCASINO-CASINOINTEGRATIONSERVICESSTATS-JOB1/target/jacoco-it.exec


      5. Configure Sonar to process IT results file

      Pretty simple! If you implemented [Unit test coverage] chapter, you just need to add one additional line to project's pom.xml file.

      <PROJECT>/pom.xml
      ?
      <properties>
        <!-- Sonar -->
        ...
        <sonar.jacoco.itReportPath>${project.basedir}/../target/jacoco-it.exec</sonar.jacoco.itReportPath>
        ...
      </properties>

      That's it.

      EXTRA: Skipping TOM and TEST modules from analysis to get a more accurate test coverage

      This section is intended to improve your test coverage results by disabling those modules that are not supposed to have tests, but are indeed analysed by Sonar and therefore, the percentage of coverage is lower than should be. The best example of this are the TOM and IT modules themselves.

      To disable the execution of one module, we just need to define an extra property in the POM file.

      pom.xml
      ?
      <sonar.skippedModules>MODULE-tom,MODULE-tests,ANY-OTHER-MODULE-NOT-USEFUL-FOR-ANALYSIS</sonar.skippedModules>

      Replace MODULE-tom with the real name of your TOM  I.e. web-api-tom, casino-integration-tests-tom, etc. See an example here.

      Troubleshooting

      Problem: When running a build, JaCoCo does not generate the unit test coverage file (jacoco.exec)

      Possible causes:

      1. Your project does not have unit tests or they are in a non-standard folder (Different than */src/test/java/). You should see the following message in your build output:

        ?
        [INFO] Skipping JaCoCo execution due to missing classes directory: $PROJECT/$MODULE/target/classes

      2. Plugin maven-surefire-plugin has some extra configuration that is preventing the JaCoCo execution. In this case, the following message should appear:

        ?
        [INFO] Skipping JaCoCo execution due to missing execution data file: $PROJECT/../target/jacoco.exec

      Solutions

      1. Write some unit tests or define the correct tests folder by adding the parameter <testSourceDirectory> to maven-surefire-plugin's configuration. Check out http://maven.apache.org/surefire/maven-surefire-plugin/test-mojo.html#testSourceDirectory for more information.
      2. Several possible solutions:

        1. If you have defined custom JVM options using the <argLine> tag, you have to make sure that it also includes ${argLine}. That variable is used by JaCoCo to send extra stuff to maven-surefire-plugin. 

          ?
                           <plugin>
                               <artifactId>maven-surefire-plugin</artifactId>
                               <configuration>
          -                        <argLine>-XX:-UseSplitVerifier</argLine>
          +                        <argLine>${argLine} -XX:-UseSplitVerifier</argLine>
                               </configuration>
                           <plugin>
        2. Maven-surefire-plugin is running in forkMode. This is not supported by JaCoCo, and it's also deprecated since version 2.14.1 (Parent POM 1.4.1 uses 2.18.1). Check out this document to see some alternatives.
